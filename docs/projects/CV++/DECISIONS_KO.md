## 요약
CV++는 우선 범용 AI 비디오 플랫폼이 아니라, 메타데이터 관측성과 신뢰성 검증에 초점을 둔 실용 제품으로 다뤄야 한다. 현재 사용자 가치의 핵심은 Hanwha RTSP 스트림이 실제로 어떤 데이터를 보내는지, 커스텀 헤더가 의도대로 적용되는지, 파싱된 메타데이터가 이후 외부 CV 모델 비교의 기반으로 쓸 만큼 신뢰할 수 있는지를 확인하는 데 있다.

## 사용자 목적
주요 사용자는 Hanwha 카메라와 NVR 환경을 다루는 개발자/운영자 역할의 SungHwan이다.

이 제품은 사용자가 다음을 할 수 있도록 도와야 한다.
- C++에서 Hanwha RTSP 스트림에 연결한다
- 커스텀 RTSP 헤더를 삽입하고 검증한다
- raw ONVIF 계열 메타데이터를 영상과 함께 확인한다
- 파싱된 객체 정보와 오버레이가 실제 스트림과 맞는지 검증한다

## MVP 범위
MVP에는 다음이 포함된다.
- 대상 Hanwha 장비에서 안정적으로 RTSP 영상을 수신하는 기능
- RTSP URL, 헤더, latency 설정의 외부 설정화
- 비디오와 메타데이터 스트림의 분리 수집
- 디버깅을 위한 raw metadata 캡처
- 사람이 검토하기 쉬운 parsed metadata 요약
- 현재 탐지 객체를 보여주는 on-screen overlay
- stale 객체가 계속 남지 않도록 하는 metadata freshness 처리

MVP는 고급 분석보다 관측성과 검증에 집중해야 한다.

## 범위 제외 항목
MVP에서는 다음을 포함하지 않는다.
- YOLO, SAM, 얼굴인식 같은 외부 AI 모델 추론
- 대규모 저장, 검색, 분석 파이프라인
- 다중 카메라 fleet 관리
- 프로덕션 클라우드 배포
- 운영자 검증을 넘는 수준의 완성형 최종 사용자 UI
- 모든 RTSP 벤더를 포괄하는 범용 지원

## 성공 기준
다음 조건을 만족하면 MVP가 성공한 것으로 본다.
- 사용자가 하드코딩 수정 없이 Hanwha RTSP 소스를 대상으로 앱을 실행할 수 있다
- 커스텀 헤더를 설정하고 실제 RTSP 흐름에서 검증할 수 있다
- raw metadata를 캡처하고 디버깅용으로 검토할 수 있다
- parsed metadata가 간결하고 사람이 확인 가능한 형태로 보인다
- overlay가 최신 metadata와 일치하고 객체가 사라지면 정상적으로 정리된다
- 코드베이스가 이후 카메라 metadata와 외부 CV 출력 비교 단계로 넘어갈 준비가 되어 있다

## 검증 UI 결정
UI는 MVP 범위 안에 포함하되, 완성형 제품 UI가 아니라 검증용 인터페이스로만 다룬다.

선호하는 MVP 인터페이스는 다음을 한 화면에서 보여주는 단순한 운영자용 데스크톱 화면이다.
- 실시간 영상
- 현재 overlay 결과
- 최근 parsed metadata 요약
- metadata freshness 또는 timestamp 상태
- raw metadata 캡처 내용 또는 최근 raw metadata 일부

이 부분에 대한 PM 결정은 명확하다. v0에서는 가장 완성도 높아 보이는 UI가 아니라, 검증 시간과 해석 모호성을 가장 많이 줄여주는 UI를 선택해야 한다. 영상, parsed 결과, raw metadata 근거를 함께 보여줄 수 있다면 그것으로 충분하다.

## 핵심 우려 사항
- 현재의 regex 기반 metadata 파싱은 구조 변화에 여전히 취약할 수 있다.
- 단일 파일 구조가 유지되면 관측성 기능 추가 시 확장이 느려질 수 있다.
- 샘플 metadata fixture가 없으면 파싱과 overlay 회귀를 잡기 어렵다.

## 대응 결정
### 1. Metadata parsing 취약성
단기 대응:
- MVP 단계에서는 regex 파싱을 유지해도 되지만, 동작이 투명하고 디버깅 가능해야 한다
- 파싱 실패와 미지원 패턴은 명시적으로 로그에 남겨야 한다
- 사용자가 raw metadata와 parsed 결과를 빠르게 대조할 수 있어야 한다

중기 대응:
- 실제 샘플이 충분히 쌓이면 더 구조적인 XML 파싱 방식으로 옮겨간다

### 2. 단일 파일 구현 리스크
단기 대응:
- 지금 당장 큰 리팩터링을 하지 않는다
- 변경 빈도가 높은 책임부터 분리한다: config loading, metadata parsing, overlay state, logging
- 각 단계마다 실행 가능한 상태를 유지한다

중기 대응:
- 하나의 큰 엔트리 파일에서 책임이 분리된 소규모 모듈 구조로 옮겨간다

### 3. 샘플 metadata fixture 부재
단기 대응:
- 실제 카메라에서 받은 raw metadata를 source of truth로 본다
- Hanwha 장비에서 대표 샘플을 소수 수집한다
- 최소한 정상 객체, 빈 장면, 객체 소멸, class pattern 변형 케이스는 보존한다

중기 대응:
- 수집된 샘플을 parser 및 overlay 회귀 확인용 fixture로 사용한다

## 권고
CV++를 Hanwha 환경 중심의 RTSP 제어 및 metadata observability MVP로 승인하는 것이 적절하다. AI 비교 기능은 metadata 수집, 파싱, 검증 신뢰도가 확보된 뒤로 미뤄야 한다.
